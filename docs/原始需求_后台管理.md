## 1. 原始输入

这个目前又合并了一个后端的端口，但是顺着我们的业务逻辑来说，我们目前缺少一个后台的管理员系统，后台管理员的系统的功能，包括一个新的/资源比如/root 后缀，其次，root 的密码需要设置，且绑定了创始人的邮箱，更换注册密码指定邮箱，然后呢，登录后台管理系统，需要能够看到用户表，其次，我们里面有个黑客松报名表，我们会设置，黑客松报名的问题，所以，后台要看到一个一个的报名情况，也就是报名的信息，然后我们的问题可能是自定义设计的，所以，我们后台要有一个黑客松的内容管理平台，然后参赛者，注册登录之后是可以完成参赛的，之后报名的所以问卷信息沉淀起来的话，我们有个 ai 自动读报名信息，并且归类，打上 tag 或者自然语义分析，首先要过滤一轮，完全乱填信息的，之后，我们可以批准通过之后，要可以定制化的定制发送邮件信息的内容，要和报名者的填写信息有关，这样才有归属感，好的，后台注册之外，要可以设置报名提交的窗口和时间，和要求，这个也是定制化的，提交完成之后，要有看板可以看项目，有 ppt,文字描述，图片等信息，不涉及视频，存在后台管理平台中，之后，用户端再开放完成之后，可以预览，并且注册登录的用户可以选择点赞作品，然后后台可以管理这些项目的展示，并且，可以根据票数，以及赛道等分类咋前端展示，大概是这些功能，先归类我的原始需求_后台管理为 doc 的 markdown文档，其次，开始规划 API 设计，规划好我们的文档结构每次完成对话之前，commit 本轮涉及修改的文件并 push

## 2. 初步归类

| 需求域 | 原始需求要点 | 当前处理 |
| --- | --- | --- |
| 后台入口 | 需要后台管理员系统，入口类似 `/root` 后缀。 | 规划 `/root/` 为后台页面入口；API 分为 `/api/root/*` 和 `/api/admin/*`。 |
| root 账号 | root 密码需要设置，绑定创始人邮箱，更换注册密码要指定邮箱。 | 规划 root 初始化、邮箱绑定、密码重置与审计。 |
| 用户管理 | 登录后台后能看到用户表。 | 规划用户列表、详情、状态、角色筛选接口。 |
| 报名管理 | 能看到每一条黑客松报名信息。 | 规划报名列表、详情、状态流转、AI 标签与审核。 |
| 自定义问卷 | 报名问题由后台自定义设计。 | 复用 `form_fields` 表，规划字段配置接口。 |
| AI 初筛 | AI 读取报名信息、归类、打 tag、自然语义分析，过滤乱填。 | 复用 `ai_tasks`、`registrations.ai_tags_json`、`registrations.ai_summary`。 |
| 审核通知 | 批准通过后定制化发送邮件，内容和报名者填写信息相关。 | 复用 `mail_templates`、`mail_logs`，规划草稿生成、预览、人工确认发送。 |
| 活动窗口 | 能设置报名提交窗口、时间、要求。 | 复用 `events.registration_open`、`registration_deadline`、`submission_deadline` 与字段规则。 |
| 项目看板 | 提交完成后后台看板查看项目、PPT、文字描述、图片等，不涉及视频。 | 复用 `projects`、`project_files`，明确 MVP 禁止视频。 |
| 前台展示 | 开放后用户端可预览，注册登录用户可以点赞作品。 | 复用 `public/projects`、`votes`，规划筛选、排序、展示状态。 |
| 展示管理 | 后台可管理项目展示，并按票数、赛道分类在前端展示。 | 规划作品发布、隐藏、精选、排序、赛道筛选接口。 |
| 工作规范 | 任务前判断体量、分析、Todo、分步更新、校验、提交推送、更新版本。 | 本轮按该流程执行。 |
