# 技术可行性与目录规划

## Java 能不能做

能做，而且适合作为实训项目。

理由：
- 实训课程已经覆盖 Java Web、MVC、MySQL、JSP、Servlet、管理员后台、用户审核等核心能力。
- 黑客松平台的主流程本质是 CRUD + 状态流转 + 文件上传 + 权限角色，非常适合 Java Web。
- AI、飞书、邮件不是必须和主流程强耦合，可以做成 service / integration 层，先 Mock 或半自动，再逐步自动化。

## 推荐实训版技术栈

- 后端：Java 17 + Servlet 3.1 + JSP
- 架构：MVC 分层
- 数据库：MySQL 8
- 数据访问：JDBC
- 前端：JSP + Bootstrap 5 + jQuery + Toastr
- 文件：本地目录备份 + 数据库存 metadata
- 飞书：先设计 FeishuSyncService，后续通过 lark-cli 或 OpenAPI 接入
- 邮件：JavaMail 或 SMTP HTTP 服务
- AI：AIService 抽象接口，后续接 OpenAI / 其他模型 API

## 后续可升级方向

当实训版闭环稳定后，可以升级为：
- Spring Boot + MyBatis
- Vue 3 / React 管理后台
- 对象存储 OSS / S3
- 飞书 Base、飞书云盘、飞书任务 / 看板深度集成
- 队伍匹配推荐、评审智能摘要、自动复盘报告

## 核心领域模型

建议先设计这些实体：

- `AdminUser`：管理员账号
- `Participant`：报名选手
- `Team`：队伍
- `TeamMember`：队伍成员关系
- `Event`：赛事 / 活动

- `RegistrationFormField`：可配置报名字段
- `Registration`：报名记录
- `AiTag`：AI 标签
- `ReviewDecision`：审核记录
- `Submission`：作品提交
- `SubmissionFile`：作品附件

- `Rating`：开放评分记录

## 状态流转

报名状态：

```text
draft -> submitted -> ai_tagged -> pending_review -> approved / rejected / waitlisted
```

队伍状态：

```text
forming -> submitted -> approved -> locked -> dissolved
```

作品状态：

```text
draft -> submitted -> needs_fix -> accepted -> published -> archived
```

开放评分状态：

```text
closed -> open -> locked -> archived
```

## 飞书策略

本地系统是流程主控，飞书是备份和协作面板。

MVP：
- 本地 MySQL 保存报名、队伍、作品、开放评分、投票、邮件日志。
- 文件本地保存一份，数据库记录路径。
- 提供 CSV / JSON 导出，便于导入飞书。

V1：
- 报名通过后同步到飞书多维表格。
- 作品附件同步到飞书云盘。
- 队伍和审核状态同步到飞书看板 / 多维表格视图。
- 系统记录每次同步状态、错误原因和重试按钮。

## 风险与规避

- 文件上传过大：限制类型和大小，MVP 不引入视频上传或视频外链。
- 飞书接口不稳定或权限复杂：先做导出，再做自动同步。
- AI 结果不可靠：AI 只给标签、摘要和建议，不自动做最终审核。
- 邮件误发：所有 AI 生成邮件必须先人工确认，再批量发送。
- 实训时间有限：先做单场赛事，不做多租户和复杂权限。
